### 99.02.1 Understand Skills And Prompts

Skills and prompts solve different customization problems.

| Customization | How it loads | Use it for this repo when |
| ------------- | ------------ | ------------------------- |
| Skill | Copilot auto-activates it from the skill `description` | You want durable knowledge about calculator setup, CSV data, PostgreSQL test containers, Blazor refactoring, or README structure |
| Prompt | You invoke it directly from Copilot Chat | You want a repeatable action such as setting up the solution, converting CSV tests, migrating to Azure, or running a quality gate |

Inspect existing calculator skills:

```powershell
Get-ChildItem .github/skills -Directory
Get-Content .github/skills/create-csv-test-dataset/SKILL.md -Head 20
Get-Content .github/skills/convert-csv-test-data-to-postgresql-container/SKILL.md -Head 20
```

Inspect existing calculator prompts:

```powershell
Get-ChildItem .github/prompts -Filter *.prompt.md
Get-Content .github/prompts/1.12.4-testing-strategy.prompt.md -Head 20
Get-Content .github/prompts/12.00.test-for-quality.prompt.md -Head 20
```

Notice the difference: skill descriptions focus on matching a topic, while
prompt descriptions and bodies define a specific action.

### 99.02.2 Create A Calculator Testing Skill

Create a practice skill that teaches Copilot finished-project testing patterns
for this calculator workspace:

````powershell
New-Item -ItemType Directory -Path .github/skills/calculator-finished-testing -Force | Out-Null

@'
---
name: calculator-finished-testing
description: Finished-project calculator testing guidance for xUnit, TestCases.csv, Testcontainers PostgreSQL, CalculatorOperations, Blazor calculator services, and solution-level validation.
---

# Calculator Finished Testing

## Purpose

Use this skill when working on tests for the completed calculator workspace at
`src/workspace/calculator-xunit-testing/`.

## Test Boundaries

* Put pure arithmetic assertions in `calculator.tests` against
  `calculator.library/CalculatorOperations.cs`.
* Keep reusable arithmetic behavior in `calculator.library`.
* Keep console input and output behavior in `calculator`.
* Keep Blazor interaction tests focused on `calculator.web` components and
  services.
* Use `TestCases.csv` as the human-editable operation dataset.
* Use Testcontainers PostgreSQL for automated test isolation, not production
  runtime configuration.

## Naming Convention

Use `MethodName_Scenario_ExpectedBehavior` for test methods.

```csharp
[Theory]
[MemberData(nameof(GetAddTestCases))]
public void Add_ValidOperands_ReturnsExpectedResult(double left, double right, double expected)

[Fact]
public void Divide_ZeroDivisor_ThrowsDivideByZeroException()
```

## Edge Cases To Cover

| Category | Examples |
| -------- | -------- |
| Arithmetic boundaries | Negative numbers, decimals, large values |
| Invalid operations | Unknown operation names, unsupported symbols |
| Division rules | Divide by zero and modulo by zero |
| CSV data | Missing file, malformed rows, unsupported operation names |
| PostgreSQL tests | Container startup, seed row count, query mapping |
| Blazor services | History updates, clearing state, theme persistence |

## Validation Commands

```powershell
dotnet build src/workspace/calculator-xunit-testing/calculator.slnx
dotnet test src/workspace/calculator-xunit-testing/calculator.slnx
```

Run the narrowest meaningful validation first. Broaden to the full solution when
shared behavior, test data loading, or service contracts change.
'@ | Out-File -FilePath .github/skills/calculator-finished-testing/SKILL.md -Encoding utf8
````

Verify the skill frontmatter:

```powershell
Get-Content .github/skills/calculator-finished-testing/SKILL.md -Head 8
```

The `description` matters because it is the part Copilot can use to decide
whether the skill is relevant before loading the full body.

### 99.02.3 Create A Calculator Test Prompt

Create a practice prompt that gives Copilot a repeatable action for generating
focused calculator tests:

````powershell
@'
---
agent: software-quality-tester
model: GPT-5.5 (copilot)
name: create-calculator-test
description: Generate focused xUnit tests for the completed calculator workspace, using CalculatorOperations, TestCases.csv, Testcontainers PostgreSQL, and Blazor service boundaries when relevant.
---

# Create Calculator Test

Generate focused xUnit tests for the completed calculator workspace.

## Instructions

1. Read the source file or behavior named by the user.
2. Inspect existing tests in
   `src/workspace/calculator-xunit-testing/calculator.tests/CalculatorTest.cs`.
3. Check whether `TestCases.csv` should be extended instead of hardcoding new
   `[InlineData]` values.
4. Keep pure arithmetic tests focused on `CalculatorOperations`.
5. Keep PostgreSQL-backed tests isolated through Testcontainers.
6. Keep Blazor service tests focused on observable service behavior.
7. Generate the smallest test change that covers the requested behavior.
8. Recommend the narrowest useful validation command.

## Naming Convention

Use `MethodName_Scenario_ExpectedBehavior`.

Examples:

* `Add_ValidOperands_ReturnsExpectedResult`
* `Divide_ZeroDivisor_ThrowsDivideByZeroException`
* `LoadTestCases_MissingCsv_ThrowsFileNotFoundException`
* `ClearHistory_WithExistingEntries_RemovesAllEntries`

## After Generating

Run or recommend one of these commands:

```powershell
dotnet test src/workspace/calculator-xunit-testing/calculator.tests/calculator.tests.csproj
dotnet test src/workspace/calculator-xunit-testing/calculator.slnx
```

Report what changed, which command ran, and any remaining risk.
'@ | Out-File -FilePath .github/prompts/create-calculator-test.prompt.md -Encoding utf8
````

Verify the prompt frontmatter:

```powershell
Get-Content .github/prompts/create-calculator-test.prompt.md -Head 8
```

The `agent` field routes the prompt to the repository's
`software-quality-tester` agent, which already understands the finished
calculator testing baseline.

### 99.02.4 Test Skill Auto-Activation

Start a new Copilot Chat session and ask a testing question that should match
the skill description:

```text
How should I test a new calculator operation that uses TestCases.csv and Testcontainers PostgreSQL?
```

The answer should mention this repository's finished test boundaries:

* `calculator.library/CalculatorOperations.cs` for arithmetic behavior
* `calculator.tests/TestCases.csv` for data-driven operation cases
* Testcontainers PostgreSQL for isolated automated database-backed tests
* `dotnet test src/workspace/calculator-xunit-testing/calculator.slnx` for broad validation

If the response is generic, mention `calculator-finished-testing`, xUnit,
`TestCases.csv`, or Testcontainers PostgreSQL explicitly and try again.

### 99.02.5 Test The Prompt

Invoke the practice prompt from Copilot Chat:

```text
/create-calculator-test for CalculatorOperations divide-by-zero behavior
```

The response should inspect existing calculator tests, preserve the naming
convention, and suggest a focused test or validation path. If your Copilot Chat
uses a different prompt invocation UI, select
`.github/prompts/create-calculator-test.prompt.md` from the prompt picker.

Try a Blazor service example:

```text

```

The response should keep UI-facing behavior in `calculator.web` and avoid
duplicating arithmetic logic in component tests.

### Key Takeaways

| Concept | File | Purpose |
| ------- | ---- | ------- |
| Skill frontmatter | `.github/skills/<name>/SKILL.md` | Describes when Copilot should auto-load the skill |
| Skill body | `.github/skills/<name>/SKILL.md` | Stores reusable domain patterns and troubleshooting guidance |
| Prompt frontmatter | `.github/prompts/*.prompt.md` | Defines the prompt name, agent, model, and description |
| Prompt body | `.github/prompts/*.prompt.md` | Defines the repeatable action Copilot should perform |
| Cleanup step | Practice skill and prompt files | Keeps temporary lab customization out of commits |

Best practices:

* Use skills for durable knowledge and conventions.
* Use prompts for repeatable actions.
* Keep descriptions specific so routing works well.
* Use repository paths and validation commands from the finished workspace.
* Remove practice files unless the team wants to maintain them.
