## Exercise 00.02 - Custom Instructions And Agents

**Module:** 00 - Explore The Copilot Workspace
**Associated prompt:** None. This is an orientation exercise.

In this lab you will practice changing Copilot behavior for this calculator
workspace by editing custom instructions and agent context. Use a short-lived
branch or be ready to revert the practice edits after the exercise.

> [!NOTE]
> Presenter pace: 4 minutes. Self-paced: 15 minutes.

References:

### Learning Objectives

* Add a focused repository-wide rule to `.github/copilot-instructions.md`.
* Create a path-specific instruction file that applies only to Blazor Razor
  components.
* Understand when an `AGENTS.md` file helps and when it duplicates existing
  documentation.
* Inspect custom agent files and identify when to use a specialized agent such
  as `sw-developer`, `software-quality-tester`, or `devops-engineer`.

### 00.01.1 Modify Repo-Wide Instructions

The repository-wide instructions file is loaded for every Copilot interaction
in this workspace. It is the right place for durable rules that should affect
calculator code, tests, prompts, and docs.

Open the file:

```bash
code .github/copilot-instructions.md
```

Add this practice section near the existing active calculator guidance:

```markdown
## Calculator Lab Practice Conventions

* Keep pure arithmetic behavior in `calculator.library`.
* Keep host-specific wiring in `calculator` or `calculator.web`.
* Keep assertions and test data setup in `calculator.tests`.
* Use Testcontainers PostgreSQL only for automated test isolation.
* Do not copy local Docker credentials or test database settings into runtime docs.
```

Then ask Copilot Chat:

```text
Where should I add a new calculator operation, and where should I test it?
```

Copilot should reference the library, console or web host, and test project
separation. If the answer does not mention those boundaries, restart the chat
session and ask again.

> [!TIP]
> Keep repo-wide instructions short and durable. If a rule only applies to one
> file type, use a path-specific instruction instead.

### 00.01.2 Create A Path-Specific Instruction

Path-specific instructions apply only when the `applyTo` glob matches files in
the current context. This is useful for Blazor component rules that should not
affect test code, Bicep files, or PowerShell scripts.

Create a Razor component instruction file for the Blazor calculator app:

```powershell
@'
---
description: "Blazor component conventions for the calculator web app"
applyTo: '**/*.razor'
---

# Blazor Component Guidelines

* Keep calculator interaction state inside component code or scoped services.
* Put reusable calculator behavior in `calculator.library` or `calculator.web/Services`.
* Dispose JavaScript interop references safely when circuits disconnect.
* Keep visible text concise so calculator controls fit on desktop and mobile screens.
* Use existing styling from `calculator.web/wwwroot` before adding new visual patterns.
'@ | Out-File -FilePath .github/instructions/blazor-components.instructions.md -Encoding utf8
```

Verify the frontmatter:

```powershell
Get-Content .github/instructions/blazor-components.instructions.md -Head 6
```

Then open a Razor component such as
`src/workspace/calculator-xunit-testing/calculator.web/Components/Pages/Home.razor`
and ask Copilot Chat:

```text
What rules should I follow before changing this calculator component?
```

Copilot should include the Blazor component guidelines only when a `.razor`
file is in context.

### 00.01.3 Add Minimal Agent Context

The source lab uses `AGENTS.md` as a place for concise project context. This
repository does not currently include a root `AGENTS.md`, because most durable
context already lives in `.github/copilot-instructions.md`, the README, prompts,
and skills.

Create `AGENTS.md` only if you have non-obvious context that Copilot cannot
infer from the codebase. For this calculator repo, a good practice example is
the boundary between test PostgreSQL and future runtime PostgreSQL:

```markdown
# Agent Context

## Architecture Decisions

### ADR-001: Testcontainers PostgreSQL Is Test Infrastructure

Status: Accepted

Context: Calculator tests seed PostgreSQL from `TestCases.csv` so xUnit can
exercise data-backed scenarios without depending on a shared database.

Decision: Testcontainers PostgreSQL remains test infrastructure. Production or
Azure runtime code must use explicit application configuration, managed secrets,
and formal migration artifacts before writing runtime data.

Consequences:

* Tests stay isolated and deterministic.
* Local Docker credentials are not copied into runtime configuration.
* Azure PostgreSQL schema, seed, rollback, and validation work is tracked as a
  separate migration concern.
```

After adding the file, ask Copilot Chat:

```text
Can I use the Testcontainers PostgreSQL setup as the production database path?
```

The expected answer should explain that Testcontainers is for automated tests,
while runtime data needs explicit application configuration and migration
artifacts.

> [!IMPORTANT]
> Do not auto-generate a large `AGENTS.md`. Keep it short and hand-written.
> Avoid copying the README or file tree into it, because Copilot can inspect
> those files directly.

### 00.01.4 Inspect Custom Agents

This repository includes custom agents under `.github/agents/`. Agent files use
YAML frontmatter for metadata and a Markdown body for role guidance.

Inspect the software developer agent:

```powershell
Get-Content .github/agents/sw-developer.agent.md -Head 40
```

Then compare it with the tester and DevOps agents:

```powershell
Get-Content .github/agents/software-quality-tester.agent.md -Head 40
Get-Content .github/agents/devops-engineer.agent.md -Head 40
```

Use these agents when the role matters:

| Scenario | Agent to consider | Why |
| -------- | ----------------- | --- |
| Add calculator operations or refactor services | `sw-developer` | Focuses on implementation in .NET-style projects |
| Generate tests for calculator, Blazor services, or PostgreSQL-backed data | `software-quality-tester` | Focuses on quality and test coverage |
| Review Azure App Service, Bicep, or workflow configuration | `devops-engineer` | Focuses on deployment and platform concerns |
| Review secrets, auth, or dependency risks | `security-engineer` | Focuses on security findings and mitigations |

Ask Copilot Chat:

```text
Which custom agent should I use to review the Azure App Service deployment files in this repo?
```

The expected answer should point toward `devops-engineer`, with
`security-engineer` as a useful follow-up for secrets and access control.

### 00.01.5 Clean Up Practice Edits

If you created practice-only instruction or context files, decide whether they
belong in the repository. Keep them only if the rules are durable and useful for
future participants.

To remove the practice files:

```powershell
Remove-Item .github/instructions/blazor-components.instructions.md -ErrorAction SilentlyContinue
Remove-Item AGENTS.md -ErrorAction SilentlyContinue
```

If you added the practice section to `.github/copilot-instructions.md`, remove
that section before committing unless your team wants to keep it.

### Success Criteria

* You can explain when to use repo-wide instructions versus path-specific
  instructions.
* You can create an `applyTo: '**/*.razor'` instruction and verify that it only
  applies to Razor component work.
* You can explain why `AGENTS.md` should be minimal and non-duplicative.
* You can identify at least two custom agents and the scenarios they support in
  this calculator repository.

### Key Takeaways

| Concept | File | Purpose |
| ------- | ---- | ------- |
| Repo-wide instructions | `.github/copilot-instructions.md` | Durable rules for all Copilot interactions in this repo |
| Path-specific instructions | `.github/instructions/*.instructions.md` | File-type or path-specific rules controlled by `applyTo` |
| Agent context | `AGENTS.md` | Optional non-obvious project context and decisions |
| Custom agents | `.github/agents/*.agent.md` | Role-specific behavior for development, testing, security, and operations |

Best practices:

* Keep instructions focused on rules and conventions.
* Keep `AGENTS.md` minimal and hand-written.
* Do not duplicate README content or obvious project structure.
* Use path-specific instructions for framework-specific guidance.
* Review practice edits before committing them.
