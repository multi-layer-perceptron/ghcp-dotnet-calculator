## Exercise 99.06.01 - GitHub Agentic Workflows

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will demonstrate a GitHub Agentic Workflow that responds to a
failed GitHub Actions run. You will manually run a simple workflow that contains
an obvious, intentionally injected failure, then observe how the
`workflow-failure-doctor` workflow is configured to diagnose that failed run.

This exercise uses two workflow files:

| File | Purpose |
| ---- | ------- |
| `.github/workflows/00.01-ghcp-workflow-with-failure.yml` | A simple manual workflow based on `00-ghcp-workflow.yml` with one intentional failure step |
| `.github/workflows/99.06.workflow-failure-doctor.md` | A GitHub Agentic Workflow source file that listens for failed runs of the demonstration workflow |

> [!NOTE]
> Presenter pace: 8 minutes. Self-paced: 20 minutes.

References:

* [GitHub Agentic Workflows](https://github.com/github/gh-aw)
* [Workflow structure](https://github.github.io/gh-aw/reference/workflow-structure/)
* [Frontmatter reference](https://github.github.io/gh-aw/reference/frontmatter/)

### Learning Objectives

* Install and verify the `gh-aw` GitHub CLI extension.
* Configure repository authentication for the selected AI engine.
* Explain how a Markdown-based GitHub Agentic Workflow differs from a standard
  YAML GitHub Actions workflow.
* Trigger an agentic workflow from a failed `workflow_run` event.
* Inspect the relationship between a workflow `name` and the matching
  `workflow_run.workflows` filter.
* Identify the intentionally failed step and propose the smallest fix.
* Describe why agentic workflows need constrained permissions, tools, and safe
  outputs.

### Prerequisites

Before starting, confirm the workflow files exist:

```powershell
Test-Path .github/workflows/00-ghcp-workflow.yml
Test-Path .github/workflows/00.01-ghcp-workflow-with-failure.yml
Test-Path .github/workflows/99.06.workflow-failure-doctor.md
```

Each command should return `True`.

The setup steps below install the GitHub Agentic Workflows tooling and configure
repository authentication. The local learning goal is still useful without
running the agent: you can inspect the trigger, failure, permissions, tools, and
output rules.

### 99.06.1 Install The `gh-aw` CLI Extension

Install the GitHub Agentic Workflows CLI extension:

```bash
gh extension install github/gh-aw
```

Verify the installation:

```bash
gh aw --version
```

Expected result: the `gh aw --version` command prints the installed extension
version.

### 99.06.2 Set Up Authentication

Choose your AI engine and add the corresponding API key as a GitHub repository
secret. For getting started, GitHub Copilot is recommended because it requires
no additional API key setup for GitHub users.

```bash
# For GitHub Copilot, recommended for getting started
gh secret set COPILOT_GITHUB_TOKEN < /path/to/token.txt

# For Anthropic Claude
gh secret set ANTHROPIC_API_KEY < /path/to/key.txt

# For OpenAI Codex
gh secret set OPENAI_API_KEY < /path/to/key.txt
```

Expected result: the selected secret is available to GitHub Actions in the
repository where you run the agentic workflow.

### 99.06.3 Understand The Two Workflow Types

Standard GitHub Actions workflows are YAML files. They describe jobs, runners,
steps, and shell commands. In this exercise, the simple failing workflow is a
standard YAML workflow:

```powershell
Get-Content .github/workflows/00.01-ghcp-workflow-with-failure.yml
```

GitHub Agentic Workflows are Markdown files with YAML frontmatter. The
frontmatter defines the trigger and guardrails. The Markdown body gives the
agent its task instructions:

```powershell
Get-Content .github/workflows/99.06.workflow-failure-doctor.md
```

The important distinction is that the Markdown body is an agent prompt. It tells
the agent how to reason about the failure, what evidence to inspect, and which
safe output to use.

### 99.06.4 Inspect The Failure Trigger

Open `.github/workflows/00.01-ghcp-workflow-with-failure.yml` and find the
workflow name:

```yaml
name: 00.01-ghcp-workflow-with-failure
```

Then open `.github/workflows/99.06.workflow-failure-doctor.md` and find the matching
`workflow_run` trigger:

```yaml
on:
  workflow_run:
    workflows: ["00.01-ghcp-workflow-with-failure"]
    types: [completed]
```

The string in `workflows` must match the standard workflow's `name`. If it does
not match exactly, `workflow-failure-doctor` will not run after the
demonstration workflow completes.

The agentic workflow should diagnose only failed runs. The output rules in the
Markdown body tell the agent to call `noop` when the triggering run did not
fail.

### 99.06.5 Inspect The Intentional Failure

In `.github/workflows/00.01-ghcp-workflow-with-failure.yml`, find the
workflow-level environment variable:

```yaml
env:
  DEMO_FAILURE: 'true'
```

Then find the final step named `intentional-failure-for-exercise-99-06`:

```yaml
- name: intentional-failure-for-exercise-99-06
```

The failure is obvious and easy to fix by design. The step exits with code `1`
when `DEMO_FAILURE` is `true`, and its error message tells you how to repair the
workflow after the diagnostic demonstration.

### 99.06.6 Compile The Agentic Workflow

Compile the GitHub Agentic Workflow source file before relying on it in GitHub
Actions:

```bash
gh aw compile .github/workflows/99.06.workflow-failure-doctor.md
```

Expected result: the compiler generates
`.github/workflows/99.06.workflow-failure-doctor.lock.yml`. Review the generated
lock workflow so you can see the full GitHub Actions workflow, pinned actions,
and security hardening produced from the Markdown source.

### 99.06.7 Commit The Workflow Files

Commit both the Markdown source and the generated lock workflow:

```bash
git add .github/workflows/99.06.workflow-failure-doctor.md
git add .github/workflows/99.06.workflow-failure-doctor.lock.yml
git commit -m "feat: add workflow failure doctor agentic workflow"
git push
```

Expected result: GitHub receives both files. The Markdown file remains the
human-authored source, and the `.lock.yml` file is the compiled GitHub Actions
workflow that runs in the repository.

### 99.06.8 Test The Agentic Workflow Setup

Confirm GitHub recognizes the compiled workflow before triggering the failing
demo workflow:

```bash
gh aw status
```

If your environment supports direct agentic workflow runs, you can also run:

```bash
gh aw run workflow-failure-doctor
```

Expected result: the agentic workflow is available through the `gh-aw` CLI and
the compiled workflow appears in the repository's Actions tab. If your
organization has not enabled direct manual `gh-aw` runs, continue with the
`workflow_run` trigger in the next section.

### 99.06.9 Run The Failing Workflow

In GitHub:

1. Open the repository on GitHub.
2. Select the **Actions** tab.
3. Select **00.01-ghcp-workflow-with-failure**.
4. Select **Run workflow**.
5. Use the current branch and start the run.
6. Wait for the run to fail in the
  `display-workflow-metadata-and-fail-intentionally` job.

Expected result:

* The `list-repository-contents` job succeeds.
* The `display-workflow-metadata-and-fail-intentionally` job fails.
* The failed step is `intentional-failure-for-exercise-99-06`.
* The failure message says to set `DEMO_FAILURE` to `false` or remove the step.

### 99.06.10 Observe The `workflow-failure-doctor` Workflow

After the failing workflow completes, GitHub evaluates the agentic workflow's
`workflow_run` trigger. `workflow-failure-doctor` is scoped to read Actions
metadata and create a safe diagnostic output:

```yaml
permissions:
  actions: read
  contents: read
  issues: read
  pull-requests: read
  checks: read
```

The workflow source also restricts its available tools and safe outputs:

```yaml
safe-outputs:
  create-issue:
  add-comment:
  noop:
tools:
  github:
    toolsets: [default, actions]
```

Expected diagnosis:

* Workflow: `00.01-ghcp-workflow-with-failure`
* Failed job: `display-workflow-metadata-and-fail-intentionally`
* Failed step: `intentional-failure-for-exercise-99-06`
* Root cause: the lab intentionally sets `DEMO_FAILURE` to `true`
* Recommended fix: set `DEMO_FAILURE` to `false` or remove the intentional
  failure step
* Validation: rerun the workflow and confirm it passes

### 99.06.11 Fix The Demonstration Failure

After you have observed the failure and reviewed the expected diagnosis, make
the smallest repair in `.github/workflows/00.01-ghcp-workflow-with-failure.yml`:

```yaml
env:
  DEMO_FAILURE: 'false'
```

Commit the repair, push it, and rerun **00.01-ghcp-workflow-with-failure** from
the Actions tab. The workflow should now pass. A passing run should not create a
new failure diagnosis because the `workflow-failure-doctor` output rules call
`noop` for successful runs.

### Key Takeaways

| Concept | Purpose |
| ------- | ------- |
| `workflow_run` | Starts follow-up automation after another workflow completes |
| Workflow `name` | The identifier used by `workflow_run.workflows` filters |
| Agentic workflow source | Markdown prompt plus YAML frontmatter guardrails |
| Safe outputs | Limits how the agent can write results back to GitHub |
| Intentional failure | Provides a predictable signal for learning the diagnostic flow |

Best practices:

* Keep demonstration failures obvious and reversible.
* Scope agentic workflow permissions to the evidence and output it needs.
* Use `noop` rules for successful runs and duplicate diagnostics.
* Make the expected diagnosis explicit in workshop labs so learners can compare
  the agent's output with known ground truth.
