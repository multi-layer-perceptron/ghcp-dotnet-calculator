## Exercise 99.07 - Copilot Coding Agent And Code Review

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will chain two platform-level agentic features: the **Copilot
Coding Agent** to repair the deliberate failure you observed in exercise 99.06,
and **Copilot Code Review** to automatically review the resulting pull request.
You will not write the fix yourself — you will write an issue, and two AI
agents handle the rest.

The target of the repair is the intentional failure injected into
`.github/workflows/00.01-ghcp-workflow-with-failure.yml`: the workflow-level
environment variable `DEMO_FAILURE: 'true'` causes the step named
`intentional-failure-for-exercise-99-06` to exit with code `1`.

> [!NOTE]
> Presenter pace: 5 minutes. Self-paced: 20 minutes.

Presenter prep: before the session, run the failing workflow so the diagnostic
issue exists, assign Copilot, and let the coding agent open a pull request that
gets auto-reviewed. When presenting, walk through the completed issue, pull
request, and review chain without waiting for the agent.

### 99.07.1 Understand Copilot Code Review

GitHub Copilot includes a built-in code reviewer that can automatically review
pull requests. No workflow files and no CI configuration — it is a native
platform feature.

| Feature | Details |
| ------- | ------- |
| Trigger | Automatic (via rulesets) or manual (assign `Copilot` as reviewer) |
| Output | Review comments — never approves or blocks merging |
| Context | Reads `copilot-instructions.md` and repository code |
| Static analysis | Integrates CodeQL, ESLint, and PMD for high-signal findings |
| Plans | Copilot Pro, Pro+, Business, and Enterprise |

This differs from the `99.06.workflow-failure-doctor` agentic workflow: the
gh-aw workflow *diagnoses failed runs* and files an issue, while Code Review
*evaluates proposed changes* on a pull request. In this lab the two meet in the
middle: the diagnosis from 99.06 becomes the issue that drives the fix, and Code
Review checks the fix before you merge it.

### 99.07.2 Configure Automatic Reviews

Configure Copilot to automatically review every pull request using repository
rulesets.

In GitHub:

1. Go to your repository, then **Settings**, then **Rules**, then **Rulesets**.
2. Select **New ruleset**, then **New branch ruleset**.
3. Name it `copilot-code-review`.
4. Under **Target branches**, add `main` (or your default branch).
5. Under **Branch rules**, check **Require a pull request before merging**.
6. Check **Request pull request review from Copilot**.
7. Optionally enable **Review new pushes** so Code Review re-reviews each push
   to the pull request branch.
8. Select **Create**.

> [!TIP]
> No ruleset? You can also manually request a review by assigning **Copilot**
> as a reviewer on any individual pull request. The ruleset just automates it.

### 99.07.3 Enable The Coding Agent

The Copilot Coding Agent turns GitHub Issues into pull requests. You describe
what you want in an issue, assign Copilot, and the agent creates a branch,
reads your codebase and custom instructions, implements the change, and opens a
pull request.

In GitHub:

1. Go to your repository, then **Settings**, then **Copilot**, then
   **Coding agent**.
2. Ensure the coding agent is **enabled**.
3. If you are in an organization, the org owner may need to enable it from
   **Organization settings**, then **Copilot**, then **Policies**.

### 99.07.4 Locate The Repair Issue

An issue should have already been created automatically in exercise 99.06: when
the failing run completed, the `99.06.workflow-failure-doctor` agentic workflow
filed a diagnostic issue through its `create-issue` safe output. It carries the
`[workflow-failure-doctor]` title prefix and the `workflow-failure` and
`automated-diagnosis` labels.

Find it:

```bash
gh issue list --label workflow-failure --label automated-diagnosis --state open
```

Open the issue and confirm it contains the failing workflow name, run URL,
failed job and step, root cause, and the recommended fix. That diagnostic issue
is your repair issue — you can assign the Coding Agent to it directly in the
next section.

If the diagnostic issue is missing, return to section 99.06.9 and rerun the
failing workflow so `99.06.workflow-failure-doctor` can file it.

> [!TIP]
> Issue quality matters. The agent-generated diagnostic issue works well
> because it contains specific evidence, file paths, and a recommended fix —
> the same clarity you would give a human developer. If the diagnosis is thin,
> add clarifying requirements as a comment before assigning the Coding Agent.

### 99.07.5 Assign The Coding Agent

In GitHub:

1. Open the diagnostic issue you located in section 99.07.4.
2. In the **Assignees** panel, select the gear icon.
3. Select **Copilot** from the list of assignees.

Or use the CLI:

```bash
gh issue edit <ISSUE_NUMBER> --add-assignee @copilot
```

What happens next:

1. The agent starts a session and posts status updates on the issue.
2. It reads the repository, including `copilot-instructions.md`.
3. It creates a branch named similar to `copilot/fix-<issue-number>`.
4. It sets `DEMO_FAILURE` to `'false'` per the issue requirements.
5. It opens a pull request linked back to the issue.

This typically takes 2 to 5 minutes. If you configured the ruleset in section
99.07.2, Code Review starts automatically as soon as the pull request opens.

### 99.07.6 Review The Automated Feedback

Your pull request now has two types of AI activity: the Coding Agent created
the pull request, and Code Review reviewed the change.

In GitHub:

1. Open the pull request and confirm **Copilot** appears under **Reviewers**.
2. Examine the diff in the **Files changed** tab. Expect a one-line change:
   `DEMO_FAILURE: 'true'` becomes `DEMO_FAILURE: 'false'`.
3. Read any Code Review comments. For a minimal change like this, feedback may
   be positive or empty — that is a useful signal too.

> [!NOTE]
> Copilot Code Review never approves or requests changes on a pull request. It
> only comments, keeping humans in control of the merge decision.

### 99.07.7 Iterate If Needed

If the Coding Agent changed more than required, or Code Review raised a
concern, comment on the pull request tagging `@copilot`:

```markdown
@copilot Please keep the change minimal: only flip DEMO_FAILURE to 'false' and
leave the intentional failure step in place.
```

The agent pushes additional commits addressing the feedback. If **Review new
pushes** is enabled, Code Review automatically re-reviews the new commits.

### 99.07.8 Merge And Validate The Repair

Once you are satisfied with the change, merge the pull request into `main`,
then rerun the previously failing workflow:

```bash
gh workflow run 00.01-ghcp-workflow-with-failure.yml --ref main
```

Watch the run complete:

```bash
gh run watch $(gh run list --workflow=00.01-ghcp-workflow-with-failure.yml --limit 1 --json databaseId --jq '.[0].databaseId')
```

Expected result: the run succeeds. The `99.06.workflow-failure-doctor` agentic
workflow evaluates the completed run, sees the conclusion is not `failure`, and
calls `noop` — no new diagnostic issue appears.

### 99.07.9 The Agentic Chain

Step back and look at the full loop you built across exercises 99.06 and 99.07:

```text
99.06: failing run ──► workflow-failure-doctor ──► diagnostic issue
99.07: repair issue ──► Coding Agent PR ──► Code Review ──► human merge
validate: rerun succeeds ──► workflow-failure-doctor calls noop
```

| Aspect | Traditional workflow | Agentic workflow (this lab) |
| ------ | -------------------- | --------------------------- |
| Diagnosis | Developer reads logs | gh-aw agent files a diagnostic issue |
| Implementation | Developer edits YAML locally | Coding Agent implements from the issue |
| Code review | Human reviewer (hours or days) | AI review in minutes plus human final review |
| Feedback loop | Push, wait, fix, push | Comment `@copilot` and the agent iterates |

The human stays in control: you write the requirements, you review the output,
and you decide to merge.

### Key Takeaways

| Concept | Details |
| ------- | ------- |
| Copilot Coding Agent | Turns GitHub Issues into pull requests with working changes |
| Copilot Code Review | Built-in AI reviewer for pull requests |
| The agentic chain | Diagnosis, issue, implementation, review, human merge |
| Custom instructions | Both agents read `copilot-instructions.md` for repository context |
| Iteration | Comment `@copilot` on pull requests to request changes |
| Human control | Code Review only comments; humans approve and merge |

Best practices:

* Feed agent-generated diagnoses (like the 99.06 issue) into repair issues so
  requirements stay grounded in evidence.
* Constrain the Coding Agent with explicit "do not modify" boundaries for
  sensitive files such as workflows.
* Keep repair pull requests minimal so Code Review feedback stays high-signal.
* Validate agent repairs by rerunning the originally failing workflow.
