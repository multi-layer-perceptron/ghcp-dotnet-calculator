### 99.08.1 Repository Indexing: How Copilot Understands Your Code

Copilot automatically builds a semantic index of your repository. This is not
keyword search — it is a meaning-based understanding of code structure, types,
relationships, and patterns.

| Feature | Details |
| ------- | ------- |
| Speed | Re-indexing is near-instant; initial indexing may take up to 60s for large repos |
| How it works | Builds a semantic index when you open the repo; refreshes as you edit |
| What it understands | Function purpose, type relationships, call hierarchies, patterns |
| Available on | All Copilot tiers with no limits on repos indexed |
| Automatic | No configuration needed — just open the repo |

Try it — ask Copilot questions about this codebase:

```text
How does the CSV-to-PostgreSQL test data flow work in the calculator tests?
```

```text
What happens when a user divides by zero? Trace the flow from the console app
to CalculatorOperations.
```

```text
Show me all the places where SquareRoot is referenced across the solution.
```

Copilot answers by searching semantically — it connects `TestCases.csv`,
`SeedFromCsv`, the `test_data` table, and the `[MemberData]` providers even when
those files never share the same keywords.

What indexing handles versus what it does not:

| Indexing handles automatically | Still needs Memory MCP |
| ------------------------------ | ---------------------- |
| Code structure and relationships | Architecture decisions ("we chose X over Y because...") |
| Type hierarchies and call graphs | Team conventions not in code |
| File contents and patterns | Insights from previous sessions |
| Project architecture (from code) | Cross-session continuity |
| Test patterns and naming | Deployment notes and meeting decisions |

### 99.08.2 Memory MCP: Persistent Knowledge Beyond Code

Some knowledge lives outside the codebase — architecture decisions, team
conventions, debugging insights. The Memory MCP server provides a persistent
knowledge graph for this kind of knowledge.

If you completed exercise 99.03, verify the Memory MCP server configuration
for the client you are using.

For VS Code:

```powershell
Get-Content .vscode/mcp.json | Select-String -Pattern '"memory"' -Context 0,5
```

For GitHub Copilot CLI:

```powershell
Get-Content .mcp.json | Select-String -Pattern '"memory"' -Context 0,5
```

Exercise 99.03 configures Memory alongside the other MCP servers used in this
workspace. Refresh both client configurations from the repository root when a
file is missing or stale:

```powershell
pwsh .github/skills/configure-mcp-servers/scripts/Set-McpServerConfiguration.ps1 -Editor All
```

| Server | Transport | Capstone use |
| ------ | --------- | ------------ |
| `azureDevOps` | Remote HTTP | Query Azure DevOps projects, work items, repositories, pull requests, and pipelines |
| `github` | Remote HTTP | Search GitHub repositories and inspect issues or pull requests |
| `microsoftLearn` | Remote HTTP | Look up current Microsoft documentation for .NET, Blazor, Azure, and GitHub Actions |
| `playwright` | Local `npx` | Validate the running Blazor calculator in a browser |
| `memory` | Local `npx` | Store and retrieve non-sensitive decisions as a knowledge graph |

VS Code reads the `servers` object in `.vscode/mcp.json`; Copilot CLI reads the
`mcpServers` object in `.mcp.json`. The `memory` entries in both configurations
set `MEMORY_FILE_PATH` to the repository-local `.memory/memory.json` file.

The Memory MCP server provides these tools:

| Tool | Purpose |
| ---- | ------- |
| `create_entities` | Create new nodes in the knowledge graph |
| `create_relations` | Link entities together |
| `add_observations` | Add facts to existing entities |
| `search_nodes` | Search the knowledge graph |
| `open_nodes` | Retrieve specific entities by name |
| `read_graph` | Read the entire knowledge graph |
| `delete_entities` | Remove entities |
| `delete_relations` | Remove relations |
| `delete_observations` | Remove specific observations |

Data model: an **entity** is a node (a thing), an **observation** is a fact
about an entity, and a **relation** is a directed edge connecting two entities.

> [!IMPORTANT]
> Store only non-sensitive project knowledge. Do not store secrets,
> credentials, connection strings, personal data, or short-lived access
> details in the knowledge graph.

### 99.08.3 Store Decisions And Context

Start with a small storage and retrieval test:

```text
Use the Memory MCP server to remember that the active calculator solution targets .NET 10 and lives under src/workspace/calculator-xunit-testing/.
```

Then retrieve the fact:

```text
Use the Memory MCP server to recall the active calculator solution target and location.
```

The response should identify `.NET 10` and
`src/workspace/calculator-xunit-testing/`. If it does not, use `MCP: List
Servers` in VS Code or `/mcp show` in Copilot CLI to confirm that the server is
started and its tools are enabled before continuing.

Use Memory MCP for knowledge that is not in the code. Ask the agent to store
this repository's decisions:

```text
Store the following in the Memory MCP knowledge graph:

Entity: CalculatorWorkshop-Decisions
Type: decisions
Observations:
- CSV file TestCases.csv is the human-editable source for arithmetic test cases
- Automated tests use Testcontainers PostgreSQL with per-run randomized credentials
- Agentic workflows compile with gh aw compile --strict before commit
- The 99.06 workflow-failure-doctor diagnoses failures and calls noop for passing runs
- DEMO_FAILURE in 00.01-ghcp-workflow-with-failure.yml toggles the intentional failure
```

Then store a relation:

```text
Store a relation in Memory MCP:
From: CalculatorWorkshop  Relation: "uses"  To: SafeOutputsPattern
Create the SafeOutputsPattern entity with type "pattern" and observation:
"Agents request writes only through declared safe outputs with max limits and noop"
```

Verify what was stored:

```text
Search the Memory MCP for "CalculatorWorkshop"
```

Store decisions, conventions, and gotchas — things that took effort to discover
and are not obvious from reading the code. Indexing already handles code
structure, so do not duplicate that.

### 99.08.4 Use Knowledge Across Sessions

The knowledge graph persists on disk, so a future session can read it.

Read the full knowledge graph:

```text
Read the entire Memory MCP knowledge graph and summarize what's stored.
```

Ask a question that uses stored knowledge:

```text
Based on what's in the Memory MCP, what testing patterns should I follow for
the calculator solution?
```

Add new observations from Exercises 99.01 through 99.08:

```text
Add these observations to the CalculatorWorkshop entity in Memory MCP:
- Has a gh-aw diagnostic workflow triggered by workflow_run failures on main
- Copilot Coding Agent repairs failures from issues; Code Review checks the PRs
- The PreToolUse hook blocks likely hardcoded secrets before write-like tools run
- The PostToolUse hook builds calculator.slnx only after create or edit operations on .cs files
```

Memory is cumulative. Each session adds knowledge, and over time the graph
becomes rich context that makes the assistant more effective.

Verify the repository-local Memory MCP file directly from the repository root:

```powershell
$MemoryFilePath = '.memory/memory.json'

if (-not (Test-Path -Path $MemoryFilePath -PathType Leaf)) {
  throw "Memory MCP file not found: $MemoryFilePath"
}

Get-Content -Path $MemoryFilePath |
  ForEach-Object { $_ | ConvertFrom-Json } |
  Select-Object type, name, entityType, from, relationType, to, observations |
  Format-List
```

The relative path resolves from the repository root in every clone. The file
uses JSON Lines format, with one entity or relation record per line.
`ConvertFrom-Json` turns each line into a structured PowerShell object, and
`Select-Object` exposes the fields needed to verify entity types, observations,
and relation endpoints. Treat this command as a read-only diagnostic. Use
Memory MCP tools to modify the graph.

### 99.08.5 Session Handoff Techniques

When you reach the end of a session (context limit, break, or task boundary),
create a handoff document so the next session can resume efficiently.

Ask Copilot to generate a handoff summary:

It captures the current Git state, completed work, remaining work, discoveries,
verification commands, and a prompt for the next session without changing your
Git history.

```powershell
pwsh .github/skills/create-session-handoff/scripts/New-SessionHandoff.ps1 `
  -CompletionSummary 'Summarize completed work.' `
  -RemainingWork 'Describe the next task.' `
  -KeyDiscovery 'Record a decision or useful finding.' `
  -VerificationCommand 'git diff --check'
```

Review the generated handoff, then create a checkpoint only when the current
changes are ready to commit:

```powershell
git add -A
git commit -m "checkpoint: session handoff $(Get-Date -Format "yyyy-MM-dd HH:mm")"
```

`HH` produces a 24-hour timestamp, without seconds. The handoff skill creates
`artifacts/session-handoffs/` when it is missing and avoids overwriting a file
created in the same minute. Use a handoff at the end of a session
(comprehensive) and a checkpoint at small milestones (lightweight).

### 99.08.7 Practice Read-Only Fleet Reviews

Fleet mode is best when each sub-agent can investigate an independent concern
without editing the same files. Use it to gather evidence first, then decide
which single follow-up change to make. Do not use it to make concurrent edits
to the calculator solution or to deploy infrastructure.

#### 99.08.7.1 Review The Active Calculator Solution

Start Copilot CLI from the repository root, enter `/fleet`, then submit this
read-only review request:

```text
Review the active calculator solution in parallel:
- Agent 1: inspect calculator.library/CalculatorOperations.cs for arithmetic edge cases.
- Agent 2: inspect calculator.tests for missing xUnit coverage and CSV test-data gaps.
- Agent 3: inspect calculator.web for state, accessibility, and error-handling risks.
Return findings only, ordered by severity. Do not modify files.
```

This split follows the project boundaries: deterministic arithmetic belongs in
`calculator.library`, test coverage belongs in `calculator.tests`, and
user-facing state belongs in `calculator.web`. Compare the findings with the
architecture and test guidance in `AGENTS.md` before acting on them.

#### 99.08.7.2 Review Azure Deployment Readiness

Start a new fleet session and submit this review request:

```text
Review this repository's Azure deployment readiness in parallel:
- Agent 1: review azure.yaml and infra/main.bicep for required parameters, naming, and deployment consistency.
- Agent 2: review GitHub workflows for secret handling, permissions, and deployment safeguards.
- Agent 3: review calculator.web configuration for production concerns such as health checks, HTTPS, and error handling.
Return a prioritized readiness checklist. Do not deploy, modify infrastructure, or access Azure resources.
```

This exercise applies the infrastructure boundaries from the Azure migration
and security exercises. Treat the fleet output as a review artifact, not proof
that deployment is safe. Validate any later infrastructure change with the
appropriate Bicep or Azure Developer CLI command before deployment.

### 99.08.8 How It All Fits Together

The Module 99 exercise series composes into one agentic development loop:

```text
99.01 instructions/agents ──► consistent AI behavior everywhere
99.02 skills/prompts      ──► reusable knowledge and commands
99.03 MCP servers         ──► Azure DevOps, GitHub, Microsoft Learn, Playwright, and Memory tools
99.04 hooks               ──► pre-write secret protection and post-edit C# build checks
99.05 orchestration       ──► parallel multi-agent coordination
99.06 agentic workflow    ──► cloud-side failure diagnosis with guardrails
99.07 coding agent + review ─► issue-to-PR repair with AI review
99.08 indexing + memory + fleet ──► code context, durable decisions, and parallel reviews
```

The workflow in practice:

1. A workflow fails and `99.06.workflow-failure-doctor` files a diagnosis.
2. You turn the diagnosis into a repair issue and assign the Coding Agent.
3. Code Review checks the pull request; you merge and validate.
4. The PreToolUse hook blocks likely hardcoded secrets; the PostToolUse hook
  builds `calculator.slnx` after C# create or edit operations.
5. A read-only fleet review separates independent findings before any shared-file change.
6. Memory MCP and a session handoff preserve the decisions and next steps for a later session.
