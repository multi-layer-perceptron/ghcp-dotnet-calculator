## Exercise 00.02 - Create A .NET Development Agent

**Module:** 00 - Explore The Copilot Workspace
**Associated prompt:** None. This is an orientation and customization exercise.

In this lab you will create a custom `.NET` development agent tailored to the
calculator workspace. The source lab creates a `dotnet-dev` agent for a sample
ContosoUniversity app. This version uses the calculator repository structure,
existing agent conventions, and staged lab workflow.

Use a short-lived branch or be ready to remove the practice agent after the
exercise.

### 00.02.1 Understand Agent Anatomy

Agent files in this repository use YAML frontmatter followed by a Markdown body.
The frontmatter describes how Copilot routes to the agent. The body gives the
agent its role, constraints, and working protocol.

Inspect an existing implementation agent:

```powershell
Get-Content .github/agents/sw-developer.agent.md -Head 40
```

Then compare it with a test-focused agent:

```powershell
Get-Content .github/agents/software-quality-tester.agent.md -Head 40
```

Notice the common fields:

| Field | Purpose |
| ----- | ------- |
| `name` | Agent identifier used by Copilot |
| `description` | Routing hint that tells Copilot when the agent is useful |
| `target` | Host environment for the custom agent |
| `model` | Preferred Copilot model for the agent |
| `handoffs` | Optional downstream agents for related work |

The Markdown body should be short enough to maintain, but specific enough to
change the agent's behavior. Avoid copying the README into the agent file.

### 00.02.2 Create The Agent File

Create a practice agent for calculator `.NET` development:

````powershell
@'
---
name: calculator-dotnet-dev
description: Specializes in .NET calculator development for this lab repository, including the console app, shared calculator library, Blazor web app, and xUnit tests.
target: vscode
model: GPT-5.5 (copilot)
handoffs:

  - label: software-quality-tester
    agent: software-quality-tester
    prompt: Generate focused tests for the calculator change.
    send: false

  - label: devops-engineer
    agent: devops-engineer
    prompt: Review application configuration and deployment impact.
    send: false
---

# Calculator .NET Developer

You are a .NET development specialist for the calculator lab repository. You
help implement and refactor calculator behavior while preserving the existing
workspace boundaries.

## When Invoked

1. Inspect the relevant files before proposing edits.
2. Keep pure arithmetic behavior in `calculator.library`.
3. Keep console input and output behavior in `calculator`.
4. Keep Blazor UI behavior in `calculator.web`.
5. Keep assertions, CSV test data, and Testcontainers setup in
   `calculator.tests`.
6. Run the narrowest relevant validation command after code changes.

## Calculator Workspace

```text
src/workspace/calculator-xunit-testing/
  calculator.slnx
  calculator/              Console workflow and input handling
  calculator.library/      Pure calculator operations
  calculator.tests/        xUnit tests, TestCases.csv, Testcontainers setup
  calculator.web/          Blazor web app and scoped services
```

## Coding Standards

* Use `#nullable enable` in C# source files.
* Add XML documentation to public members.
* Validate input with early returns or clear exceptions.
* Keep calculator operations deterministic and side-effect free.
* Do not move Testcontainers PostgreSQL into runtime application code.

## Development Commands

```powershell
dotnet build src/workspace/calculator-xunit-testing/calculator.slnx
dotnet test src/workspace/calculator-xunit-testing/calculator.slnx
dotnet run --project src/workspace/calculator-xunit-testing/calculator.web
```

## Review Checklist

* `dotnet build` succeeds for the solution.
* Relevant tests pass or the remaining failure is explained.
* Shared logic remains in `calculator.library`.
* UI code does not duplicate arithmetic logic.
* No secrets or local database credentials are committed.
'@ | Out-File -FilePath .github/agents/calculator-dotnet-dev.agent.md -Encoding utf8
````

Verify the file was created:

```powershell
Get-Content .github/agents/calculator-dotnet-dev.agent.md -Head 12
```

You should see `name: calculator-dotnet-dev` in the YAML frontmatter.

> [!TIP]
> If Copilot does not recognize the new agent, start a new chat session or
> reload the VS Code window so the custom agent list refreshes.

### 00.02.3 Consider Tools And MCP Servers

The source lab adds optional MCP server context. This repository does not
currently include `.copilot/mcp-config.json`, so the practice agent should work
without MCP server configuration.

Confirm whether an MCP configuration exists:

```powershell
Test-Path .copilot/mcp-config.json
```

If the command returns `False`, leave MCP configuration out of the practice
agent. The existing repository context, instructions, skills, prompts, and files
are enough for this exercise.

If your team later adds `.copilot/mcp-config.json`, update the agent only when
it needs a specific external capability such as documentation lookup or project
memory. Keep the dependency explicit in frontmatter and document why the server
is needed.

### 00.02.4 Test The Agent

Open Copilot Chat and invoke the new agent:

```text
@calculator-dotnet-dev What are the project boundaries in this calculator workspace?
```

The answer should describe the console app, shared library, test project, and
Blazor web app under `src/workspace/calculator-xunit-testing/`.

Try a development planning question:

```text
@calculator-dotnet-dev What would I need to change to add a modulo operation to the calculator?
```

The answer should mention updating shared arithmetic behavior, adding tests, and
checking any console or Blazor UI paths that expose available operations.

Then try a validation question:

```text
@calculator-dotnet-dev Which command should I run after changing CalculatorOperations.cs?
```

The expected answer should prefer a focused `dotnet test` or solution-level
validation command for `src/workspace/calculator-xunit-testing/calculator.slnx`.

### 00.02.5 Clean Up Or Keep The Agent

This exercise creates a practice agent. Keep it only if the team wants a durable
calculator-specific development role in the repository.

To remove the practice agent:

```powershell
Remove-Item .github/agents/calculator-dotnet-dev.agent.md -ErrorAction SilentlyContinue
```

To keep it, review the description and body for durable repo guidance rather
than one-time lab instructions.

### Success Criteria

* You can explain the frontmatter and Markdown body of a custom agent file.
* You created `.github/agents/calculator-dotnet-dev.agent.md` for practice.
* The agent references the calculator workspace boundaries accurately.
* The agent can answer a repository-specific development planning question.
* You decided whether to remove or keep the practice agent.

### Key Takeaways

| Concept | Purpose |
| ------- | ------- |
| Custom agent frontmatter | Helps Copilot route requests to the agent |
| Agent body | Defines the role, context, workflow, and guardrails |
| Handoffs | Suggest related agents for testing, DevOps, or security review |
| MCP servers | Optional external tool access when the repo has matching configuration |
| Practice cleanup | Keeps the repository free of temporary customization files |

Best practices:

* Give agents narrow, role-specific responsibilities.
* Include repository-specific paths and validation commands.
* Avoid duplicating the README or full project tree.
* Add MCP servers only when the agent needs that capability.
* Remove practice agents before committing unless they are useful long term.
