### 99.08.1 Repository Indexing: How Copilot Understands Your Code

Copilot automatically builds a semantic index of your repository. This is not
keyword search — it is a meaning-based understanding of code structure, types,
relationships, and patterns.

| Feature | Details |
| ------- | ------- |
| Speed | Re-indexing is near-instant; initial indexing may take up to 60s for large repos |
| How it works | Builds a semantic index when you open the repo; refreshes as you edit |
| What it understands | Function purpose, type relationships, call hierarchies, patterns |
| Available on | All Copilot tiers with no limits on repos indexed |
| Automatic | No configuration needed — just open the repo |

Try it — ask Copilot questions about this codebase:

```text
How does the CSV-to-PostgreSQL test data flow work in the calculator tests?
```

```text
What happens when a user divides by zero? Trace the flow from the console app
to CalculatorOperations.
```

```text
Show me all the places where SquareRoot is referenced across the solution.
```

Copilot answers by searching semantically — it connects `TestCases.csv`,
`SeedFromCsv`, the `test_data` table, and the `[MemberData]` providers even when
those files never share the same keywords.

What indexing handles versus what it does not:

| Indexing handles automatically | Still needs Memory MCP |
| ------------------------------ | ---------------------- |
| Code structure and relationships | Architecture decisions ("we chose X over Y because...") |
| Type hierarchies and call graphs | Team conventions not in code |
| File contents and patterns | Insights from previous sessions |
| Project architecture (from code) | Cross-session continuity |
| Test patterns and naming | Deployment notes and meeting decisions |

### 99.08.2 Memory MCP: Persistent Knowledge Beyond Code

Some knowledge lives outside the codebase — architecture decisions, team
conventions, debugging insights. The Memory MCP server provides a persistent
knowledge graph for this kind of knowledge.

If you completed exercise 99.03, verify the Memory MCP server configuration:

```powershell
Get-Content .copilot/mcp-config.json | Select-String -Pattern '"memory"' -Context 0,5
```

If it is not configured yet, add it:

```json
"memory": {
  "type": "local",
  "command": "npx",
  "args": ["-y", "@modelcontextprotocol/server-memory"],
  "tools": ["*"]
}
```

The Memory MCP server provides these tools:

| Tool | Purpose |
| ---- | ------- |
| `create_entities` | Create new nodes in the knowledge graph |
| `create_relations` | Link entities together |
| `add_observations` | Add facts to existing entities |
| `search_nodes` | Search the knowledge graph |
| `open_nodes` | Retrieve specific entities by name |
| `read_graph` | Read the entire knowledge graph |
| `delete_entities` | Remove entities |
| `delete_relations` | Remove relations |
| `delete_observations` | Remove specific observations |

Data model: an **entity** is a node (a thing), an **observation** is a fact
about an entity, and a **relation** is a directed edge connecting two entities.

### 99.08.3 Store Decisions And Context

Use Memory MCP for knowledge that is not in the code. Ask the agent to store
this repository's decisions:

```text
Store the following in the Memory MCP knowledge graph:

Entity: CalculatorWorkshop-Decisions
Type: decisions
Observations:
- CSV file TestCases.csv is the human-editable source for arithmetic test cases
- Automated tests use Testcontainers PostgreSQL with per-run randomized credentials
- Agentic workflows compile with gh aw compile --strict before commit
- The 99.06 workflow-failure-doctor diagnoses failures and calls noop for passing runs
- DEMO_FAILURE in 00.01-ghcp-workflow-with-failure.yml toggles the intentional failure
```

Then store a relation:

```text
Store a relation in Memory MCP:
From: CalculatorWorkshop  Relation: "uses"  To: SafeOutputsPattern
Create the SafeOutputsPattern entity with type "pattern" and observation:
"Agents request writes only through declared safe outputs with max limits and noop"
```

Verify what was stored:

```text
Search the Memory MCP for "CalculatorWorkshop"
```

Store decisions, conventions, and gotchas — things that took effort to discover
and are not obvious from reading the code. Indexing already handles code
structure, so do not duplicate that.

### 99.08.4 Use Knowledge Across Sessions

The knowledge graph persists on disk, so a future session can read it.

Read the full knowledge graph:

```text
Read the entire Memory MCP knowledge graph and summarize what's stored.
```

Ask a question that uses stored knowledge:

```text
Based on what's in the Memory MCP, what testing patterns should I follow for
the calculator solution?
```

Add new observations from this module:

```text
Add these observations to the CalculatorWorkshop entity in Memory MCP:
- Has a gh-aw diagnostic workflow triggered by workflow_run failures on main
- Copilot Coding Agent repairs failures from issues; Code Review checks the PRs
- Post-tool-use hooks verify the .NET build after C# edits
```

Memory is cumulative. Each session adds knowledge, and over time the graph
becomes rich context that makes the assistant more effective.

### 99.08.5 Automatic Learning With Continuous Learning

Memory MCP stores knowledge explicitly — you tell it what to remember. The
continuous-learning-v2 skill learns automatically from your sessions.

Examine the skill:

```powershell
Get-Content .github/skills/continuous-learning-v2/SKILL.md | Select-Object -First 50
```

The v2 architecture uses `preToolUse` and `postToolUse` hooks — the same hook
types you created in exercise 99.04. Every tool call is observed, and patterns
become atomic "instincts" with confidence scores:

| Feature | v1 (session-end) | v2 (instinct-based) |
| ------- | ---------------- | ------------------- |
| When | sessionEnd hook | preToolUse/postToolUse hooks (every action) |
| Granularity | Full skills | Atomic instincts with confidence scores |
| Confidence | None | 0.3 (tentative) to 0.9 (near-certain) |
| Evolution | Direct to skill | Instincts cluster into skills, commands, or agents |

Check the hook wiring in this repository:

```powershell
Get-Content .github/hooks/default.json | Select-String -Pattern 'observe'
```

Three layers of knowledge work together: repository indexing is automatic code
understanding, Memory MCP is your explicit decision store, and continuous
learning is your automatic pattern extractor.

### 99.08.6 Session Handoff Techniques

When you reach the end of a session (context limit, break, or task boundary),
create a handoff document so the next session can resume efficiently.

Ask Copilot to generate a handoff summary:

```text
Generate a session handoff document with:
- Current git state (branch, recent commits, modified files)
- What was completed in this module and what remains
- Key discoveries and decisions
- Verification commands
- A fresh-context prompt block for the next session
```

For lightweight intermediate saves, use a checkpoint instead — a short commit
or stash marker with a descriptive label:

```powershell
git add -A
git commit -m "chore: checkpoint after module 99 capstone"
```

Use a handoff at the end of a session (comprehensive) and a checkpoint at small
milestones (lightweight).

### 99.08.7 Advanced Copilot CLI Features

The features covered in this module — agents, skills, hooks, MCP, agentic
workflows — are building blocks. The Copilot CLI adds capabilities that amplify
them:

| Feature | Command | Purpose |
| ------- | ------- | ------- |
| LSP integration | `/lsp` | Semantic code intelligence: go-to-definition, references, diagnostics |
| Context visualization | `/context` | See how full your context window is |
| Context compaction | `/compact` | Summarize history to free context space |
| Session info | `/session` | View session info and workspace summary |
| Session export | `/share` | Export session to Markdown or a gist |
| Parallel sub-agents | `/fleet` | Experimental parallel sub-agent execution |
| Autopilot mode | `Shift+Tab` | Agent works until the task is complete |

The GitHub MCP server also enables remote semantic search without cloning:

```text
Search the GitHub repo github/gh-aw for how safe-outputs create-issue grouping works
```

### 99.08.8 How It All Fits Together

The module 99 exercises compose into one agentic development loop:

```text
99.01 instructions/agents ──► consistent AI behavior everywhere
99.02 skills/prompts      ──► reusable knowledge and commands
99.03 MCP servers         ──► external tools (PostgreSQL, memory, docs)
99.04 hooks               ──► lifecycle guardrails (build checks, observation)
99.05 orchestration       ──► parallel multi-agent coordination
99.06 agentic workflow    ──► cloud-side failure diagnosis with guardrails
99.07 coding agent + review ─► issue-to-PR repair with AI review
99.08 memory + learning   ──► knowledge that persists and compounds
```

The workflow in practice:

1. A workflow fails and `99.06.workflow-failure-doctor` files a diagnosis.
2. You turn the diagnosis into a repair issue and assign the Coding Agent.
3. Code Review checks the pull request; you merge and validate.
4. Hooks verify builds after edits; observations feed continuous learning.
5. Memory MCP stores the decisions so the next session starts smarter.
