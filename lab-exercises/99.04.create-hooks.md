## Exercise 99.04 - Create Hooks

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will create a practice post-tool-use hook that verifies the
finished calculator solution still builds after a C# file is edited. The hook is
tailored to `src/workspace/calculator-xunit-testing/calculator.slnx` and the
completed calculator workspace.

Use this exercise only after completing the current exercise sequence from
[Exercise 00.01 - Explore Copilot Configuration Files](00.01.explore-copilot-config-files.md)
through [Exercise 03.02 - Comprehensive Quality Gate](03.02.comprehensive-quality-gate.md), so the
solution exists under `src/workspace/calculator-xunit-testing/`.

This repository now includes a committed hook suite in `.github/hooks/` and
`scripts/hooks/`, covering observation logging, secret scanning, doc blocking,
and a .NET build check. This exercise walks you through building the .NET build
hook yourself so you understand how the committed hook works. You can keep your practice hook or remove it after you finish.

### 99.04.1 Understand The Build Hook

A post-tool-use hook can run after Copilot edits or creates a file. For this
calculator repo, the useful guardrail is narrow: after C# file edits, run
`dotnet build` for the active solution and report a concise result.

The hook should not run after every tool call. It should exit without output
unless the tool operation edited or created a `.cs` file.

| Hook input | Expected behavior |
| ---------- | ----------------- |
| `TOOL_NAME=edit`, `FILE_PATH=...CalculatorOperations.cs` after adding `SquareRoot` | Build `calculator.slnx` |
| `TOOL_NAME=edit`, `FILE_PATH=...CalculatorTest.cs` after adding square root tests | Build `calculator.slnx` |
| `TOOL_NAME=create`, `FILE_PATH=...HistoryService.cs` | Build `calculator.slnx` |
| `TOOL_NAME=read`, `FILE_PATH=...CalculatorOperations.cs` | Exit without building |
| `TOOL_NAME=edit`, `FILE_PATH=README.md` | Exit without building |

### 99.04.2 Create The Hook Script

Create a hook script directory and a PowerShell build hook:

````powershell
New-Item -ItemType Directory -Path scripts/hooks -Force | Out-Null

@'
# Post-tool-use hook: verify the calculator solution after C# edits.

$ErrorActionPreference = 'Stop'

$toolName = $env:TOOL_NAME
$filePath = $env:FILE_PATH

if ($toolName -ne 'edit' -and $toolName -ne 'create') {
    exit 0
}

if ([string]::IsNullOrWhiteSpace($filePath) -or $filePath -notlike '*.cs') {
    exit 0
}

$repoRoot = Resolve-Path (Join-Path $PSScriptRoot '..\..')
$solutionPath = Join-Path $repoRoot 'src\workspace\calculator-xunit-testing\calculator.slnx'

if (-not (Test-Path $solutionPath)) {
    Write-Host "Calculator solution not found: $solutionPath"
    exit 1
}

$buildOutput = & dotnet build $solutionPath --nologo --verbosity quiet 2>&1
if ($LASTEXITCODE -ne 0) {
    Write-Host "BUILD FAILED after editing $filePath"
    Write-Host ''
    $buildOutput | Select-Object -Last 20 | ForEach-Object { Write-Host $_ }
    Write-Host ''
    Write-Host 'Fix the build errors before continuing.'
    exit 1
}

Write-Host "Build succeeded after editing $filePath"
'@ | Out-File -FilePath scripts/hooks/post-tool-use-dotnet-build.ps1 -Encoding utf8
````

Verify the script exists:

```powershell
Get-Content scripts/hooks/post-tool-use-dotnet-build.ps1 -Head 20
```

The script uses the same solution path as the main lab validation command, so it
checks the console app, shared library, tests, and Blazor web app together.

### 99.04.3 Register The Hook

Create the hook configuration directory and register the script as a
`postToolUse` hook:

```powershell
New-Item -ItemType Directory -Path .github/hooks -Force | Out-Null

@'
{
  "postToolUse": [
    {
      "type": "command",
      "powershell": "./scripts/hooks/post-tool-use-dotnet-build.ps1",
      "timeoutSec": 30,
      "comment": "Verify calculator solution build after editing C# files"
    }
  ]
}
'@ | Out-File -FilePath .github/hooks/default.json -Encoding utf8
```

Verify the hook registration:

```powershell
Get-Content .github/hooks/default.json | Select-String dotnet-build
```

Set a timeout that leaves enough room for `dotnet build` to run. If the hook
times out, Copilot may not receive useful build output.

### 99.04.4 Add SquareRoot, Tests, And Trigger The Build Hook

Add a small C# change that gives the hook a realistic rebuild target. The skill
below handles the library edit idempotently, so learners can rerun the same
request without creating duplicate methods. In Copilot Chat, ask:

```text
Use the add-square-root-operation skill to update the calculator library for lab 99.04.
```

If your Copilot session was already open before this skill was added, reload
skills first, then run the request again.

The skill should leave this method near the other operation methods in
`src/workspace/calculator-xunit-testing/calculator.library/CalculatorOperations.cs`:

```csharp
/// <summary>
/// Calculates the square root of the specified operand.
/// </summary>
/// <param name="operand">The operand value.</param>
/// <returns>The square root of the operand.</returns>
/// <exception cref="ArgumentOutOfRangeException">Thrown when the operand is negative.</exception>
public static double SquareRoot(double operand)
{
    if (operand < 0)
    {
        throw new ArgumentOutOfRangeException(nameof(operand), "Cannot calculate the square root of a negative number.");
    }

    return Math.Sqrt(operand);
}
```

This method is intentionally small: it gives the hook a real calculator library
edit and gives the test project a focused behavior to verify.

Next, the test project needs matching SquareRoot coverage through the existing
PostgreSQL-backed `[MemberData]` pattern. The skill below handles the fragile
edits for the test methods, public data provider, and CSV rows idempotently, so
learners can rerun the same request without duplicating tests or CSV data. In
Copilot Chat, ask:

```text
Use the add-square-root-test-coverage skill to update the calculator test project for lab 99.04.
```

If your Copilot session was already open before this skill was added, reload
skills first, then run the request again.

The skill should leave these tests near the other operation tests in
`src/workspace/calculator-xunit-testing/calculator.tests/CalculatorTest.cs`:

```csharp
[Theory]
[MemberData(nameof(GetSquareRootTestCases))]
public void SquareRoot_WithNonNegativeOperand_ReturnsSquareRoot(
    double operand,
    double ignoredSecondOperand,
    double expectedResult)
{
    _ = ignoredSecondOperand;

    var actualResult = CalculatorOperations.SquareRoot(operand);

    Assert.Equal(expectedResult, actualResult, precision: 10);
}

[Fact]
public void SquareRoot_WithNegativeOperand_ThrowsArgumentOutOfRangeException()
{
    Assert.Throws<ArgumentOutOfRangeException>(() => CalculatorOperations.SquareRoot(-1));
}
```

The `ignoredSecondOperand` value keeps this test compatible with the existing
three-value `MemberData` shape. Assigning it to discard makes the intent
explicit and avoids analyzer warnings about unused theory parameters.

The skill should leave this provider near the other `Get*TestCases` methods:

```csharp
public static IEnumerable<object[]> GetSquareRootTestCases() => GetTestCases("SquareRoot");
```

It should also leave these rows in
`src/workspace/calculator-xunit-testing/calculator.tests/TestCases.csv`. The
calculator uses one operand for square root, so the second operand stays `0` to
preserve the existing four-column CSV structure:

```csv
0,0,SquareRoot,0
4,0,SquareRoot,2
9,0,SquareRoot,3
2,0,SquareRoot,1.4142135623730951
```

The test project seeds its PostgreSQL test data from `TestCases.csv` at runtime,
so these CSV rows are the associated PostgreSQL test data update. If you are
using a long-running local PostgreSQL database instead of the Testcontainers
test path, rerun the repository's CSV-to-PostgreSQL seed workflow after changing
the CSV.

The `CalculatorOperations.cs` and `CalculatorTest.cs` edits are both C# edits,
so either one should trigger the build hook in a real post-tool-use flow. The
CSV edit updates the data that the PostgreSQL-backed tests load, but it should
not trigger the C# build hook by itself.

Because this exercise adds `SquareRoot` to a `.cs` file, the hook should trigger
and run `dotnet build`. Simulate the environment variables a hook receives after
the `CalculatorOperations.cs` edit:

```powershell
$env:TOOL_NAME = 'edit'
$env:FILE_PATH = 'src/workspace/calculator-xunit-testing/calculator.library/CalculatorOperations.cs'
.\scripts\hooks\post-tool-use-dotnet-build.ps1
```

You should see `Build succeeded` if the finished solution currently builds.
Run the test project as a follow-up check to verify the new `SquareRoot` rows
are loaded from `TestCases.csv` into the PostgreSQL-backed test data path:

```powershell
dotnet test src/workspace/calculator-xunit-testing/calculator.tests/calculator.tests.csproj --nologo --verbosity quiet
```

If the build fails because `Math` is unavailable in the file, add `using
System;` at the top of `CalculatorOperations.cs` or qualify the call as
`System.Math.Sqrt(operand)`, then rerun the hook.

As a contrast, verify the same hook script skips the build for non-C# file
events:

```powershell
$env:TOOL_NAME = 'edit'
$env:FILE_PATH = 'README.md'
.\scripts\hooks\post-tool-use-dotnet-build.ps1
```

This command simulates a triggered hook event for a Markdown edit. The hook
script should inspect `FILE_PATH`, recognize that `README.md` is not a C# file,
and exit without running `dotnet build`. The `SquareRoot` edit above is the path
that should trigger the actual rebuild.

Clear the test environment variables when you are done:

```powershell
Remove-Item Env:\TOOL_NAME -ErrorAction SilentlyContinue
Remove-Item Env:\FILE_PATH -ErrorAction SilentlyContinue
```

### 99.04.5 Decide Whether To Keep The Hook

Build hooks can be useful, but they also add latency after edits. Keep this hook
only if the team wants automatic build verification after C# changes.

Good fits for this calculator repo:

* Refactoring `calculator.library/CalculatorOperations.cs`.
* Updating Blazor services under `calculator.web/Services`.
* Changing public contracts that affect tests or UI code.
* Teaching participants how post-edit validation can catch build regressions.

Poor fits:

* Running after documentation, Markdown, Bicep, YAML, or prompt edits.
* Replacing focused test runs when behavior changes.
* Committing hook automation without team review.

To remove only your practice copies, delete the specific files you created.
Do not remove the whole `.github/hooks/` or `scripts/hooks/` directories — they
now contain the committed hook suite used by later exercises, including the
continuous-learning observation hooks referenced in exercise 99.08:

```powershell
git checkout -- .github/hooks/default.json scripts/hooks/post-tool-use-dotnet-build.ps1
```

### Key Takeaways

| Concept | Purpose |
| ------- | ------- |
| `postToolUse` | Runs after Copilot completes a tool operation |
| `TOOL_NAME` | Identifies the tool operation that triggered the hook |
| `FILE_PATH` | Identifies the file affected by the operation |
| Build hook | Detects compile regressions soon after C# edits |
| Cleanup step | Prevents temporary automation from becoming accidental policy |

Best practices:

* Keep hooks narrow and fast.
* Filter by tool name and file extension before running expensive commands.
* Use repository-specific solution paths.
* Print concise failure output that helps Copilot repair the problem.
* Review hook automation before committing it.
