## Exercise 99.05 - Multi-Agent Orchestration

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will use parallel multi-agent orchestration to extend the
Azure-hosted Blazor calculator with two additional calculator modes: binary and
hexadecimal. Each mode supports the simple operations `+`, `-`, `*`, and `/`.

This exercise uses parallel implementation agents: one agent works on binary
mode in one isolated worktree, while another works on hexadecimal mode in a
second isolated worktree. You then integrate both results and run one final
review.

Use this exercise only after completing the current exercise sequence from
[Exercise 00.01 - Explore Copilot Configuration Files](00.01.explore-copilot-config-files.md)
through [Exercise 03.03 - Cleanup And Reset](03.03-cleanup-and-reset.md), so the
solution exists under `src/workspace/calculator-xunit-testing/` and includes the
Blazor web app.

### 99.05.1 Design The Parallel Orchestration Flow

Multi-agent orchestration lets you coordinate several specialized agents. For
this calculator lab, use a parallel flow:

```text
User Request
    |
    v
You as orchestrator
    |
    +--> Worktree A: binary-mode agent
    |       |
    |       +--> Add binary input, display, tests, and validation notes
    |
    +--> Worktree B: hex-mode agent
    |       |
    |       +--> Add hexadecimal input, display, tests, and validation notes
    |
    v
Integration branch
    |
    +--> Merge both workstreams
    +--> Resolve shared Blazor UI and CalculatorService conflicts
    +--> Run build and tests
    +--> Request final security review
    |
    v
Summary returned to user
```

The parallel split works because binary mode and hexadecimal mode have similar
requirements but can be implemented independently when each agent has its own
working directory. The integration step is where shared files such as
`CalculatorService.cs`, `CalculatorKeypad.razor`, `Home.razor`, and
`CalculatorServiceTests.cs` are reconciled.

Use the sequential `lab-orchestrator` agent in the next section as a small
practice artifact, then switch to parallel worktrees for the actual binary and
hexadecimal implementation work.

### 99.05.3 Define The Feature Contract

Before starting parallel work, write down the common contract that both agents
must follow.

Target behavior:

* The Blazor calculator keeps the existing decimal mode as the default.
* Users can switch between decimal, binary, and hexadecimal modes.
* Binary mode accepts only `0` and `1` as numeric input.
* Hexadecimal mode accepts `0` through `9` and `A` through `F` as numeric input.
* Binary and hexadecimal modes support only `+`, `-`, `*`, and `/`.
* Binary and hexadecimal mode calculations parse input as signed integers and
  format results in the active base.
* Division by zero keeps the existing error behavior.
* Non-whole division results in binary or hexadecimal mode show a clear error
  instead of silently truncating.
* History records show the expression and result in the active mode.
* Existing decimal behavior, history replay, keyboard input, and Azure App
  Service deployment files continue to work.

Useful implementation surfaces:

| Area | Files to inspect |
| ---- | ---------------- |
| Calculator state and operations | `src/workspace/calculator-xunit-testing/calculator.web/Services/CalculatorService.cs` |
| Display formatting | `src/workspace/calculator-xunit-testing/calculator.web/Services/CalculatorFormatter.cs` |
| Main Blazor page | `src/workspace/calculator-xunit-testing/calculator.web/Components/Pages/Home.razor` |
| Keypad buttons | `src/workspace/calculator-xunit-testing/calculator.web/Components/Calculator/CalculatorKeypad.razor` |
| Web app styling | `src/workspace/calculator-xunit-testing/calculator.web/wwwroot/app.css` |
| Service tests | `src/workspace/calculator-xunit-testing/calculator.tests/CalculatorServiceTests.cs` |
| Azure hosting config | `azure.yaml`, `infra/main.bicep` |

### 99.05.4 Prepare Parallel Worktrees

Create two isolated worktrees from the repository root. Each worktree gets its
own branch and working directory:

```powershell
git worktree add ../ghcp-calculator-binary -b feature/calculator-binary-mode
git worktree add ../ghcp-calculator-hex -b feature/calculator-hex-mode
```

Open one terminal for each worktree:

```powershell
cd ../ghcp-calculator-binary
```

```powershell
cd ../ghcp-calculator-hex
```

In each terminal, confirm the solution still builds before assigning work:

```powershell
dotnet build src/workspace/calculator-xunit-testing/calculator.slnx --nologo --verbosity quiet
```

> [!IMPORTANT]
> Do not run both agents in the same working directory. Parallel agents sharing
> one checkout can overwrite each other's edits or create confusing Git index
> state. Worktrees keep the file systems separate while sharing repository
> history.

### 99.05.5 Run Binary And Hex Agents In Parallel

Start one Copilot session in each worktree. Use the software developer agent
from the lab-orchestrator mapping.

In the binary worktree, give the agent this request:

```text
@sw-developer-agent Implement binary calculator mode for the Blazor calculator.

Scope:
- Work only in this binary-mode worktree.
- Preserve existing decimal calculator behavior.
- Add a mode selector that includes Decimal, Binary, and Hexadecimal, but only implement Binary behavior in this branch.
- Binary mode accepts digits 0 and 1.
- Binary mode supports +, -, *, and /.
- Parse binary operands as signed integers and format binary results in base 2.
- Keep division-by-zero behavior consistent with the current calculator.
- Show a clear error for non-whole binary division results.
- Add or update CalculatorService tests for binary addition, subtraction, multiplication, division, invalid digits, and non-whole division.
- Run dotnet build and dotnet test for src/workspace/calculator-xunit-testing/calculator.slnx.

Files to inspect first:
- src/workspace/calculator-xunit-testing/calculator.web/Services/CalculatorService.cs
```

### 99.05.6 Integrate The Parallel Work

After both worktrees pass their local build and tests, return to the main
repository checkout and create an integration branch:

```powershell
cd ../ghcp-dotnet-calculator
git switch -c feature/calculator-base-modes
```

Merge the binary work first:

```powershell
git merge feature/calculator-binary-mode
```

Then merge the hexadecimal work:

```powershell
git merge feature/calculator-hex-mode
```

Expect conflicts in shared Blazor and service files. Resolve them by keeping one
combined implementation that supports all three modes:

* Decimal mode preserves the original behavior.
* Binary mode and hexadecimal mode share parsing and formatting helpers where
  practical.
* `CalculatorKeypad.razor` exposes the right digit buttons for the active mode.
* `Home.razor` contains one clear mode selector.
* `CalculatorServiceTests.cs` includes decimal regression tests plus binary and
  hexadecimal mode tests.

Run focused validation after resolving conflicts:

```powershell
dotnet build src/workspace/calculator-xunit-testing/calculator.slnx --nologo --verbosity quiet
dotnet test src/workspace/calculator-xunit-testing/calculator.slnx --nologo --verbosity quiet
```

If the Azure App Service deployment files were touched, validate that the app
still has a normal Azure Developer CLI entry point:

```powershell
Get-Content azure.yaml
Test-Path infra/main.bicep
```

### Key Takeaways

| Concept | Purpose |
| ------- | ------- |
| Parallel orchestration | Splits independent feature work across agents that run at the same time |
| Worktree | Gives each agent an isolated checkout while sharing Git history |
| Feature contract | Keeps parallel agents aligned on behavior before implementation starts |
| Integration branch | Combines parallel work and resolves shared-file conflicts |
| Final review | Checks the merged behavior after all parallel work is reconciled |

Best practices:

* Split parallel work by behavior, not by random files.
* Give every agent the same shared contract and separate mode-specific scope.
* Keep shared UI decisions visible in integration notes.
* Validate each worktree before merging it.
* Run final build and tests after conflict resolution, not only in the parallel
  branches.
