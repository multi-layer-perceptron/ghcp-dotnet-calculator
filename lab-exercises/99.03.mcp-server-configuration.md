## Exercise 99.03 - MCP Server Configuration

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will use the `configure-mcp-servers` skill to configure optional
Model Context Protocol (MCP) servers for this completed calculator workspace.
Use this exercise only after completing the current exercise sequence from
[Exercise 00.01 - Explore Copilot Configuration Files](00.01.explore-copilot-config-files.md)
through [Exercise 03.03 - Cleanup And Reset](03.03-cleanup-and-reset.md), so the
solution exists under `src/workspace/calculator-xunit-testing/`.

The skill owns the client-specific configuration details. For VS Code, it writes
`.vscode/mcp.json` with a top-level `servers` object. For generic Copilot-style
clients, it can write `.copilot/mcp-config.json` with a top-level `mcpServers`
object. This keeps editor detection and schema differences out of the lab body.

### 99.03.1 Examine Current MCP Configuration

MCP servers extend Copilot with tools outside the repository. They can retrieve
official documentation, fetch web content, query internal systems, or persist
project knowledge depending on the server you configure.

Check whether this repository already has the VS Code workspace MCP
configuration file:

```powershell

```

If the file exists, inspect it before changing it:

```powershell

```

If you also see `.copilot/mcp-config.json`, treat it as a client-specific or
practice artifact. It does not replace `.vscode/mcp.json` for VS Code MCP
activation.

The skill configures two servers by default:

* `fetch` for retrieving public web pages or raw documentation content during a
  lab.

> Treat MCP configuration as executable tool configuration. Review server
> packages, commands, URLs, and exposed tools before committing the file.

### 99.03.3 Understand Server Types

This workspace uses two common MCP server styles:

* `microsoft-learn` uses `http` to look up official Microsoft documentation for
  .NET, Blazor, Azure App Service, Bicep, and Azure Developer CLI tasks.
* `fetch` uses `stdio` to run through `npx` and retrieve public web or raw
  documentation content when a lab references an external page.

HTTP servers connect to a remote endpoint. Stdio servers run a local command on
your machine and communicate through standard input and output. Local stdio
servers require the runtime named in the command. In this example, `fetch`
requires Node.js and `npx`.

### 99.03.4 Activate And Trust The Servers

After the skill writes `.vscode/mcp.json`, start or reload the MCP servers from
VS Code:

1. Open the Command Palette.
2. Run `MCP: List Servers`.
3. Select `microsoft-learn` or `fetch`.
4. Choose `Start`, `Enable`, or `Show Output` as needed.
5. Accept the trust prompt if VS Code asks whether you trust the workspace MCP
   configuration.
6. Open Copilot Chat and use **Configure Tools** to confirm the MCP tools are
   enabled.

If the servers do not appear, run `Developer: Reload Window`, then check
`MCP: List Servers` again. VS Code stores server enablement and trust state
separately from `.vscode/mcp.json`, so a valid file can still require a manual
start, enable, or trust action.

### 99.03.6 Compare Non-VS Code MCP Clients

MCP is portable, but configuration files are client-specific. The portable parts
are the server endpoint, command, and arguments. The file path and wrapper keys
vary by client.

Use the `configure-mcp-servers` skill for the maintained client coverage notes.
The lab intentionally automates only the VS Code workspace path and a generic
`mcpServers` reference shape. Do not copy unverified JSON paths for Claude,
Codex, Gemini, or another harness into this exercise unless you have checked
that client's current MCP documentation.

GitHub Copilot documentation currently describes MCP setup and usage for these
IDE families:

* Visual Studio Code
* Visual Studio
* JetBrains IDEs
* Xcode
* Eclipse

Those IDEs can expose MCP configuration through UI-driven setup flows even when
the JSON shape resembles `servers`. Follow the current IDE-specific setup
screen rather than assuming the VS Code workspace file path applies.

For VS Code, use `.vscode/mcp.json`:

```json
{
  "servers": {
    "microsoft-learn": {
      "type": "http",
      "url": "https://learn.microsoft.com/api/mcp"
    },
    "fetch": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-fetch"]
    }
  }
}
```

Other MCP clients might use a top-level `mcpServers` object instead:

```json
{
  "mcpServers": {
    "fetch": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-fetch"]
    }
  }
}
```

Before copying configuration to another client, check that client's MCP
documentation for three things:

* The expected configuration file location
* Whether the top-level key is `servers`, `mcpServers`, or another name
* Whether the client supports remote `http` servers, local `stdio` servers, or
  both
* Whether the client expects manual JSON editing, command-line registration, or
  an in-product MCP configuration screen

### 99.03.7 Connect MCP Usage To This Project

Use MCP only when it adds information that the repository itself does not have.

Good fits for this calculator lab:

* Checking current Azure App Service guidance before changing `infra/main.bicep`
  or `azure.yaml`
* Looking up current Blazor hosting or component guidance before changing
  `calculator.web`
* Fetching external lab references or raw README files when creating new
  exercises
* Comparing Microsoft Learn deployment guidance with the repository's existing
  Azure migration assessment

Poor fits:

* Replacing local code search for files already in this repository
* Storing secrets, local connection strings, or personal credentials
* Adding broad tool access without a concrete lab need
* Committing MCP configuration without team review

### 99.03.8 Clean Up Or Keep The Configuration

Decide whether `.vscode/mcp.json` belongs in this repository. Keep it only if
the team wants a durable MCP setup for future participants and has reviewed the
server commands, URLs, and exposed tools.

To remove the VS Code MCP configuration:

```powershell
Remove-Item .vscode/mcp.json -ErrorAction SilentlyContinue
```

If `.vscode` only existed for this exercise and is now empty, remove the folder:

```powershell
Remove-Item .vscode -Recurse -Force -ErrorAction SilentlyContinue
```

### Key Takeaways

* MCP servers add external tools to Copilot beyond local repository context.
* Skills are the right place for reusable MCP setup logic and editor-specific
  configuration differences.
* `.vscode/mcp.json` is the VS Code workspace MCP configuration file.
* `servers` is the VS Code top-level MCP server object.
* GitHub Copilot MCP setup is documented for VS Code, Visual Studio, JetBrains
  IDEs, Xcode, and Eclipse, but setup flows and paths can differ.
* HTTP servers connect to remote MCP endpoints without local processes.
* Stdio servers run local commands and communicate over standard input and
  output.
* Trust and enablement control whether VS Code starts and exposes configured
  MCP servers.
* Cleanup prevents temporary tool configuration from becoming accidental repo
  policy.

Best practices:

* Start without MCP when local files answer the question.
* Add MCP servers only for specific documentation, retrieval, or integration
  needs.
* Review local server commands before running them.
* Avoid storing secrets or credentials in MCP configuration.
* Use `MCP: List Servers` and **Configure Tools** to verify activation.
* Check client-specific MCP documentation before reusing configuration outside
  VS Code.
