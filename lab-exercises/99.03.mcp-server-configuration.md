## Exercise 99.03 - MCP Server Configuration

**Module:** 99 - Finished Project Customization
**Associated prompt:** None. This is a finished-project customization exercise.

In this lab you will use the `configure-mcp-servers` skill to configure optional
Model Context Protocol (MCP) servers for this completed calculator workspace.
Use this exercise only after completing the current exercise sequence from
[Exercise 00.01 - Explore Copilot Configuration Files](00.01.explore-copilot-config-files.md)
through [Exercise 03.03 - Final Validation And Handoff](03.03-final-validation-and-handoff.md), so the
solution exists under `src/workspace/calculator-xunit-testing/`.

The skill owns the client-specific configuration details. For VS Code, it writes
`.vscode/mcp.json` with a top-level `servers` object. For GitHub Copilot CLI,
it can write `.mcp.json` with a top-level `mcpServers` object. This keeps
editor detection and schema differences out of the lab body.

This workspace configures five MCP servers in `.vscode/mcp.json`:

* `azureDevOps`: hosted Azure DevOps MCP server over `http` at
  `https://mcp.dev.azure.com/autocloudarc-mcaps` for projects, work items,
  repositories, pull requests, and pipelines.
* `github`: hosted GitHub MCP server over `http` at
  `https://api.githubcopilot.com/mcp/` for repository, issue, and pull request
  tools.
* `microsoftLearn`: hosted Microsoft Learn MCP server over `http` at
  `https://learn.microsoft.com/api/mcp` for official documentation lookup.
* `playwright`: local `stdio` server launched with
  `npx -y @playwright/mcp@latest` for browser automation and UI validation.
* `memory`: local `stdio` server launched with
  `npx -y @modelcontextprotocol/server-memory` for knowledge-graph memory.

`npx`. Check whether both commands are available before relying on those
servers:

```powershell

Get-Command npx -ErrorAction SilentlyContinue
```

Both commands must resolve. A common Windows failure mode is a broken Node
install where `npx` shims exist on PATH but `node.exe` does not, which produces
`"node" is not recognized` errors when VS Code starts the server. If that
happens, install Node.js to a neutral path such as `C:\tools\nodejs` (a
portable ZIP install works without admin rights) and add that folder to the
PATH environment variable, then restart VS Code so new processes pick up the
change. The setup script skips Playwright and memory when these prerequisites
are missing.

### 99.03.2 Configure MCP Servers With The Skill

Use the `configure-mcp-servers` skill to create the MCP configuration for the
active editor. In Copilot Chat, ask:

```text
Use the configure-mcp-servers skill to configure Azure DevOps, GitHub, Microsoft Learn, Playwright, and memory MCP servers for this workspace.
```

The bundled script defaults to this repository's Azure DevOps organization,
`autocloudarc-mcaps`. Learners using another organization must provide their
own organization value. Use only the organization URL segment:

```powershell
pwsh .github/skills/configure-mcp-servers/scripts/Set-McpServerConfiguration.ps1 -AzureDevOpsOrganization your-organization
```

The server URL follows this pattern:

```text
https://mcp.dev.azure.com/{organization}
```

For example, `your-organization` produces
`https://mcp.dev.azure.com/your-organization`. Do not use the Azure DevOps
portal URL pattern `https://dev.azure.com/{organization}` as the MCP endpoint.

For a deterministic local run, execute the bundled script from the repository
root:

```powershell
pwsh .github/skills/configure-mcp-servers/scripts/Set-McpServerConfiguration.ps1
```

If editor detection is ambiguous, pass the target explicitly:

```powershell
pwsh .github/skills/configure-mcp-servers/scripts/Set-McpServerConfiguration.ps1 -Editor VSCode
```

```powershell
pwsh .github/skills/configure-mcp-servers/scripts/Set-McpServerConfiguration.ps1 -Editor CopilotGeneric
```

The `CopilotGeneric` target writes the GitHub Copilot CLI file at `.mcp.json`.
Verify it before starting a CLI session that depends on MCP:

```powershell
Test-Path .mcp.json
Get-Content .mcp.json | Select-String -Pattern '"mcpServers"'
```

Playwright and memory are skipped automatically when no Node.js and `npx`
launcher is found. The three hosted `http` servers are still written.

The skill configures these servers by default:

* `azureDevOps` for projects, work items, repositories, pull requests, and
  pipelines in the organization provided through `-AzureDevOpsOrganization`.
* `github` for repository, issue, and pull request tools through the hosted
  GitHub MCP endpoint.
* `microsoftLearn` for official Microsoft documentation related to .NET,
  Blazor, Azure App Service, Azure Developer CLI, Bicep, and GitHub Actions.
* `playwright` for browser automation and UI validation against the running
  Blazor calculator.
* `memory` for storing and retrieving non-sensitive project facts and
  relationships as a knowledge graph.

> [!IMPORTANT]
> Treat MCP configuration as executable tool configuration. Review server

### 99.03.3 Understand Server Types

This workspace uses both common MCP server styles:

* `azureDevOps`, `github`, and `microsoftLearn` use `http` to connect to hosted
  MCP endpoints.
  No local runtime is required, but VS Code may prompt for authentication or
  trust on first use.
* `playwright` and `memory` use `stdio` to run local commands through `npx`.
  Local stdio servers require the runtime named in the command, so they need
  Node.js and `npx` resolvable on PATH.

HTTP servers connect to a remote endpoint. Stdio servers run a local command on
your machine and communicate through standard input and output.

> [!NOTE]
> The remote Azure DevOps MCP server is in public preview and uses Microsoft
> Entra ID OAuth. When VS Code prompts you, sign in with an account that belongs
> to the organization and has access to the resources you query.

### 99.03.4 Activate And Trust The Servers

After the skill writes `.vscode/mcp.json`, start or reload the MCP servers from
VS Code:

1. Open the Command Palette.
2. Run `MCP: List Servers`.
3. Select a server such as `azureDevOps`, `github`, `microsoftLearn`,
  `playwright`, or `memory`.
4. Choose `Start`, `Enable`, or `Show Output` as needed.
5. Accept the trust prompt if VS Code asks whether you trust the workspace MCP
   configuration.
6. Complete interactive authentication if prompted. Hosted servers such as
  Azure DevOps and GitHub may require a browser-based sign-in before their
  tools become available.
7. Open Copilot Chat and use **Configure Tools** to confirm the MCP tools are
   enabled.

If the servers do not appear, run `Developer: Reload Window`, then check
`MCP: List Servers` again. VS Code stores server enablement and trust state
separately from `.vscode/mcp.json`, so a valid file can still require a manual
start, enable, or trust action.

### 99.03.6 Compare Non-VS Code MCP Clients

MCP is portable, but configuration files are client-specific. The portable parts
are the server endpoint, command, and arguments. The file path and wrapper keys
vary by client.

Use the `configure-mcp-servers` skill for the maintained client coverage notes.
The lab intentionally automates only the VS Code workspace path and the GitHub
Copilot CLI root `.mcp.json` path. Do not copy unverified JSON paths for
Claude, Codex, Gemini, or another harness into this exercise unless you have
checked that client's current MCP documentation.

GitHub Copilot documentation currently describes MCP setup and usage for these
IDE families:

* Visual Studio Code
* Visual Studio
* JetBrains IDEs
* Xcode
* Eclipse

Those IDEs can expose MCP configuration through UI-driven setup flows even when
the JSON shape resembles `servers`. Follow the current IDE-specific setup
screen rather than assuming the VS Code workspace file path applies.

For VS Code, use `.vscode/mcp.json`. An abbreviated configuration shape looks
like this:

```json
{
  "servers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/"
    },
    "azureDevOps": {
      "type": "http",
      "url": "https://mcp.dev.azure.com/autocloudarc-mcaps"
    },
    "microsoftLearn": {
      "type": "http",
      "url": "https://learn.microsoft.com/api/mcp"
    },
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"]
    }
  }
}
```

For GitHub Copilot CLI, use `.mcp.json`. An abbreviated CLI configuration uses
a top-level `mcpServers` object:

```json
{
  "mcpServers": {
    "azureDevOps": {
      "type": "http",
      "url": "https://mcp.dev.azure.com/your-organization",
      "tools": ["*"]
    }
  }
}
```

In Copilot CLI, verify visibility with:

```powershell
copilot mcp list
```

Inside an interactive CLI session, use `/mcp show` or `/mcp reload` after
editing `.mcp.json`.

Before copying configuration to another client, check that client's MCP
documentation for three things:

* The expected configuration file location
* Whether the top-level key is `servers`, `mcpServers`, or another name
* Whether the client supports remote `http` servers, local `stdio` servers, or
  both
* Whether the client expects manual JSON editing, command-line registration, or
  an in-product MCP configuration screen

### 99.03.8 Clean Up Or Keep The Configuration

Decide whether `.vscode/mcp.json` belongs in this repository. Keep it only if
the team wants a durable MCP setup for future participants and has reviewed the
server commands, URLs, and exposed tools.


### Key Takeaways

* MCP servers add external tools to Copilot beyond local repository context.
* Skills are the right place for reusable MCP setup logic and editor-specific
  configuration differences.
* `.vscode/mcp.json` is the VS Code workspace MCP configuration file.
* `servers` is the VS Code top-level MCP server object.
* `.mcp.json` is the GitHub Copilot CLI workspace MCP configuration file.
* `mcpServers` is the GitHub Copilot CLI top-level MCP server object.
* GitHub Copilot MCP setup is documented for VS Code, Visual Studio, JetBrains
  IDEs, Xcode, and Eclipse, but setup flows and paths can differ.
* HTTP servers such as `azureDevOps`, `github`, and `microsoftLearn` connect to
  remote MCP endpoints without local processes.
* The Azure DevOps endpoint uses
  `https://mcp.dev.azure.com/{organization}`; learners replace the placeholder
  with their own organization value.
* The `playwright` stdio server runs locally through `npx` and requires Node.js
  resolvable on PATH.
* A broken Node install with `npx` shims but no `node.exe` causes
  `"node" is not recognized` MCP startup failures; fix it with a neutral-path
  Node install (for example `C:\tools\nodejs`) added to PATH.
* Trust and enablement control whether VS Code starts and exposes configured
  MCP servers.
* Cleanup prevents temporary tool configuration from becoming accidental repo
  policy.

Best practices:

* Start without MCP when local files answer the question.
* Add MCP servers only for specific documentation, retrieval, or integration
  needs.
* Review local server commands before running them.
* Avoid storing secrets or credentials in MCP configuration.
* Use `MCP: List Servers` and **Configure Tools** to verify activation.
* Check client-specific MCP documentation before reusing configuration outside
  VS Code.
